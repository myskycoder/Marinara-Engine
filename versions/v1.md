# v1 — NPC Pipeline Cleanup

> Обзор реализованного плана по очистке и стабилизации NPC asset pipeline.
> Дата: 2026-04-27.

## Цели

Решить три класса проблем, накопившихся в NPC-конвейере (auto-материализация
персонажей в Game-режиме + генерация аватаров и спрайтов):

1. **Race conditions** — несколько асинхронных писателей в `chat.metadata`
   (auto-материализатор, scene-wrap, ручной regenerate, character-tracker)
   делали `read → JSON.parse → mutate → JSON.stringify → write` без
   сериализации, и одна ветка пайплайна молча затирала результаты другой.
2. **Дублирование** — 4–5 копий нормализации/слугификации NPC-имён, 3 копии
   lazy-loader'а для `sharp`, две копии трёхфазного fallback-резолвинга
   аватаров, отсутствие request-coalescing для генерации спрайтов.
3. **UX-регрессии** — `useNpcAssetWatcher` молотил с фиксированным интервалом,
   не учитывал visibility-state и обрывался по жёсткому 60-секундному
   таймеру; default game-mode агенты молча возвращались в чат, даже если
   пользователь их сознательно убрал.

## Реализованные пункты плана

### #7 — Per-chat mutex + atomic metadata RMW API

**Файл:** `packages/server/src/services/storage/chats.storage.ts`

- Добавлен модуль-локальный `chatLocks: Map<string, Promise<unknown>>`.
- Хелпер `withChatLock(id, fn)`: новые задачи становятся в цепочку после
  существующего promise, после settle убирают свою запись из мапы только
  если за это время никто не встал в очередь следом (защита от growth).
- Новый публичный метод `chats.updateMetadataWithMerge(id, mergeFn)`:
  - читает свежий `chat.metadata` под мьютексом;
  - вызывает `mergeFn(current)`, который может вернуть либо новый объект
    metadata (он будет записан), либо `null` (no-op, запись пропускается);
  - всё под единым per-chat-мьютексом — `read → mutate → write` атомарны
    относительно других вызывающих этот же API.
- Гранулярность per-chat: разные чаты не блокируют друг друга.

**Мигрировано на новый API:**

| Файл                                                 | Что писало                                            |
| ---------------------------------------------------- | ----------------------------------------------------- |
| `services/game/npc-materializer.service.ts`          | `patchNpcAvatarUrl`, `patchNpcSpriteStatus`, `regenerateNpcAssets`, `materializeGameNpcs` |
| `routes/avatars.routes.ts`                           | `POST /npc/:chatId` — patching `gameNpcs[].avatarUrl` |
| `routes/game.routes.ts`                              | `party-turn` reputation, `scene-wrap` library-resolve, `scene-wrap` background persist, `/generate-assets` avatar persist |

Старые `updateMetadata(id, full)` намеренно оставлены в путях, не пересекающихся с
NPC-полями (createGame, follow-up session bootstrap, mass batch-апдейты в
других доменах) — это явно очерченный скоп плана.

### #1 — Shared NPC name utilities

**Создан:** `packages/shared/src/utils/npc-name.ts` (+ экспорт из
`packages/shared/src/index.ts`).

Три функции:

- `npcNameKey(name)` — Unicode-aware (`\p{L}\p{N}`) канонический ключ для
  сравнений: `"Алёша  -- Попович!"` → `"алёша попович"`.
- `isSameNpcName(left, right)` — `true`, если совпадают `npcNameKey`-варианты
  ИЛИ их compact-варианты (без пробелов). Покрывает кейсы типа
  `"Mr.Smith"` vs `"MrSmith"`.
- `slugifyForFs(name, opts)` — синхронный slug для имён файлов. NFKD →
  strip combining marks → ASCII lowercase → `[^a-z0-9]+ → -` → trim. Если
  результат пустой (кириллица, эмоджи), используется fallback вида
  `<prefix>-<hash[0..10]>`. По умолчанию — небольшой 64-битный
  не-криптографический hash; server-callers инжектят `sha1HexLegacy` для
  обратной совместимости с уже-записанными файлами.

**Создан:** `packages/server/src/services/game/npc-name-server.ts` —
тонкая server-обёртка, реэкспортирует shared API и добавляет
`sha1HexLegacy(input)` через `node:crypto.createHash("sha1")`. Shared package
не зависит от `node:crypto`, поэтому SHA-1 живёт на стороне сервера.

**Удалены/заменены копии в:**

- `services/game/npc-materializer.service.ts` (локальные `slugifyNpcName`,
  `normalizeNameKey`, `compactNameKey`, `isSameNpcName`).
- `routes/avatars.routes.ts` (`slugifyName`, `nameKey`).
- `routes/game.routes.ts` (inline `slugifyNpcName`).
- `services/game/game-asset-generation.ts` — `safeName` теперь тонкая
  обёртка над `slugifyForFs(..., {prefix:"s", hashHex: sha1HexLegacy})`,
  помечена `@deprecated`.
- `client/src/lib/game-character-name-match.ts` — `normalizeCharacterName`
  делегирует в `npcNameKey` после client-specific ASCII diacritic strip.

### #2 — NPC avatar resolver

**Создан:** `packages/server/src/services/game/npc-avatar-resolver.ts`.

`resolvePresentCharacterAvatars(presentChars, ctx)` — единственная точка для
трёхфазного fallback-резолвинга:

1. **Character cards** (`knownCharacterAvatars: Map<string, string>`) —
   персонажи из чат-карточек.
2. **`gameNpcs[].avatarUrl`** — материализованные Game-mode NPC.
3. **Legacy filesystem** — `<DATA_DIR>/avatars/npc/<chatId>/<slug>.png` для
   чатов, существовавших до Auto NPC Materializer.

Мутирует `presentChars[i].avatarPath` in-place, чтобы оба call-site
(`chats.routes.ts` GET game-state и `generate.routes.ts` character-tracker
branch) могли скармливать массив прямо в SSE-события и в БД-строку без
дополнительной мапы.

**Подключён в:** `routes/chats.routes.ts:320`, `routes/generate.routes.ts:5430`.

### #4 — Watcher с exponential backoff

**Файл:** `packages/client/src/hooks/use-game.ts` (`useNpcAssetWatcher`).

- Цепочка `setTimeout` вместо `setInterval` — каждый tick пере-шедулит
  следующий с растущей задержкой.
- Backoff: `3 → ~4.8 → ~7.7 → ~12.3 → ~19.7 → ~31.5 → ~50.4 → 60` секунд
  (cap 60s), множитель 1.6.
- **Visibility-aware pause**: при `document.visibilityState !== "visible"`
  пропускаем `qc.invalidateQueries`, шедулим короткий keep-alive 5s.
- **Reset on focus**: `visibilitychange` со значением `"visible"` сбрасывает
  `attempt = 0`, очищает текущий timer и шедулит немедленный poll —
  пользователь, вернувшийся во вкладку, видит обновление сразу.
- Жёсткий ceiling (старые `MAX_ATTEMPTS = 12`) убран сознательно: при
  60-секундной cadence нагрузка пренебрежимо мала, а долгие
  image-провайдеры легко убегают за 60s × 12.

### #5 — Sprite request coalescing

**Файл:** `packages/server/src/services/game/npc-sprite-generation.service.ts`.

Зеркало `inFlightNpcPortraits` из `game-asset-generation.ts`:

- `inFlightNpcSprites: Map<string, Promise<boolean>>` — ключ `spriteId`.
- Disk pre-check (`existsSync(full_idle.png)`) **снаружи** мапы — это
  дешёвый и частый short-circuit, ему не нужна синхронизация.
- `try/finally` гарантирует cleanup записи после settle.
- Экспортирован peek-helper `getInFlightNpcSprite(spriteId)` симметрично
  `getInFlightNpcPortrait` для будущей координации между путями.

Проблема, которую это закрывает: одновременный клик «Regenerate sprite» +
работающий auto-pipeline (или две одновременные turns) тратили два полных
image-API вызова на один и тот же NPC и гонкались на запись PNG.

### #6 — `gameModeAutoSeeded` flag

Добавлено поле `gameModeAutoSeeded?: boolean` в `ChatMetadata`
(`packages/shared/src/types/chat.ts`).

Семантика: `true` после того, как для чата хотя бы один раз были
автозасеяны default game-mode агенты (`character-tracker`, `world-state`,
`persona-stats`). После этого сервер **никогда** молча не дописывает
defaults обратно в `activeAgentIds`.

**Три точки установки:**

| Где                                                 | Когда                                                                  |
| --------------------------------------------------- | ---------------------------------------------------------------------- |
| `routes/game.routes.ts` — `POST /game/create`       | первый bootstrap — флаг ставится сразу                                 |
| `routes/game.routes.ts` — follow-up session         | если `prev.gameModeAutoSeeded !== true`, делаем разовый legacy-merge и ставим флаг; иначе копируем `activeAgentIds` вербатим |
| `client/.../ChatSettingsDrawer.tsx` — кнопка миграции | «Add Game Mode Agents» merge-mutate, паруется с `gameModeAutoSeeded: true` |

Это снимает регрессию, когда удалённый пользователем `world-state` молча
возвращался при создании следующей session.

## Архитектурные эффекты

- **Единый источник истины** для NPC-имени: `npcNameKey` живёт в
  `@marinara-engine/shared`, поэтому client (matcher диалогов) и server
  (materializer, аватары, журнал) теперь не могут разойтись в
  нормализации.
- **Атомарность RMW** на `chat.metadata` — теперь любые писатели,
  использующие `updateMetadataWithMerge`, гарантированно видят последнюю
  версию metadata и не затирают друг друга. SQLite row-locks этого
  гарантировать не могли — JS-часть `read → parse → mutate` оставалась
  не-сериализованной до этой правки.
- **Обратная совместимость файлов на диске** не сломана: server-side
  callers `slugifyForFs` всё ещё используют `prefix:"s"` + `sha1HexLegacy`,
  поэтому уже-сгенерированные `<DATA_DIR>/avatars/npc/<chatId>/s-<sha1[0..10]>.png`
  остаются находимыми.
- **Меньше дублирующих API-вызовов** — coalescing для sprite-генерации
  закрывает последний крупный пробел; portrait-coalescing был добавлен
  раньше.

## Известные риски и follow-up'ы

Не блокеры для merge, отмечено в ходе ревью:

1. **Старый `updateMetadata` (full overwrite) ещё используется в ~30 местах**
   — большая часть из них в `generate.routes.ts` и `game.routes.ts` для
   полей вне NPC-домена. Если будущие фичи начнут писать `gameNpcs` через
   `updateMetadata`, mutex'а уже не хватит. Кандидаты на постепенную
   миграцию: `generate.routes.ts:5746`, `game.routes.ts:6896`.
2. **Holder-pattern в `npc-materializer.service.ts`** добавил ~10–15 строк
   boilerplate для проброса значений из callback'а наружу. Можно
   улучшить, расширив сигнатуру `updateMetadataWithMerge` до tuple-возврата
   `{ row, result }`.
3. **Микро-гонка в `useNpcAssetWatcher`** — между `tick`-ом и
   `visibilitychange` теоретически может на короткий момент существовать
   два таймера. React Query дедуплицирует invalidate, реальный impact
   нулевой, но при необходимости легко закрыть флагом `inTick`.
4. **`resolvePresentCharacterAvatars` без compact-fallback'а** — старая
   inline-логика в `chats.routes.ts` использовала compact ASCII
   normalization. Новая через `npcNameKey` сохраняет пробелы, что чуть
   строже. Для нормальных имён эквивалентно, для краевых
   («Mr.Smith» vs «MrSmith») — пропустит. В `isSameNpcName` compact-fallback
   присутствует, при необходимости можно перевести resolver на
   двухпроходный поиск.
5. **Тестов нет** — `pnpm test` определён, scope пустой. Хорошие
   кандидаты на покрытие в первую очередь:
   - race-test на `withChatLock` (10 параллельных инкрементов → 10);
   - `npcNameKey` / `isSameNpcName` / `slugifyForFs` примеры с кириллицей.
